# Generate Dependabot Config

Generate or update `.github/dependabot.yaml` by scanning all `package.json` files in the project, classifying dependencies into concern-based groups, and merging with any existing configuration.

## Process

### Step 1: Collect Dependencies

Read all `package.json` files in the project (exclude `node_modules/`, `.yarn/`). Collect every unique dependency name from `dependencies`, `devDependencies`, and `peerDependencies`. Exclude `workspace:` references (internal packages).

### Step 2: Read Existing Config

If `.github/dependabot.yaml` (or `.yml`) exists, read it fully. Parse:

1. **Top-level settings** (version, schedule, labels, commit-message, ignore, etc.) — preserve exactly
2. **Existing groups** — these inform classification (see Step 3)
3. **Manual markers** — comments that lock patterns or groups in place (see Comment Convention below)

### Step 3: Classify Dependencies into Groups

Classification uses three sources, in priority order:

1. **Manual markers** — `# manual:` patterns and `# manual-group:` groups are never modified
2. **Existing groups** — if a dependency already matches a pattern in an existing group, keep it there. Existing groups are the primary classification signal. Only add new deps or create new groups for uncovered deps.
3. **Auto-classification rules** — for dependencies not covered by (1) or (2), use the rules below

#### Auto-Classification Rules

Match dependencies to groups using these rules (first match wins):

| Group Name | Patterns | Matching Strategy |
|---|---|---|
| `react-related` | `react`, `react-dom`, `@types/react`, `@types/react-dom` | exact + types |
| `graphql-related` | `graphql`, `graphql-*`, `@graphql-codegen/*`, `@graphql-yoga/*`, `@apollo/*` | scope + prefix |
| `vite-related` | `vite`, `vite-*`, `@vitejs/*` | exact + prefix + scope |
| `vitest-related` | `vitest`, `vitest-*`, `@vitest/*` | exact + prefix + scope |
| `tailwind-related` | `tailwindcss`, `@tailwindcss/*`, `tailwind-*`, `tw-animate-css` | scope + prefix + exact |
| `typescript-related` | `typescript`, `@tsconfig/*`, `ts-essentials` | exact + scope |
| `formatting-linting-related` | `eslint`, `eslint-*`, `prettier`, `@typescript-eslint/*` | exact + prefix + scope |
| `testing-related` | `@testing-library/*`, `supertest`, `testcontainers`, `happy-dom` | scope + exact |
| `urql-related` | `urql`, `@urql/*` | exact + scope |
| `git-hooks-related` | `husky`, `lint-staged`, `@commitlint/*` | exact + scope |

**Organization-scoped packages** (e.g., `@abinnovision/*`) should be grouped by their functional concern, not by org scope. For example, `@abinnovision/eslint-config-*` belongs in `formatting-linting-related`.

**Ungrouped dependencies** that don't match any rule stay ungrouped (individual PRs). Do NOT create a catch-all group — small individual PRs are fine for one-off deps like `rimraf`, `rxjs`, `zustand`, `dotenv`, etc.

### Step 4: Merge and Write

1. Preserve all top-level config (version, schedule, labels, commit-message, ignore, open-pull-requests-limit, etc.)
2. For each group:
   - If marked `# manual-group:` → keep entirely as-is (don't add or remove patterns)
   - If it has `# manual:` on specific patterns → keep those patterns, but auto-manage the rest
   - Otherwise → regenerate patterns from current dependencies
3. Add new groups for newly classified deps
4. Remove empty auto-generated groups (all their deps were removed from package.json)
5. Sort groups alphabetically for consistency
6. Write to `.github/dependabot.yaml`

## Comment Convention

Two marker types for preserving manual overrides across regenerations:

### `# manual:` — Lock a Single Pattern

```yaml
groups:
  testing-related:
    patterns:
      - "vitest"
      - "@testing-library/*"
      # manual: msw pinned here for test-dep coordination
      - "msw"
```

The `# manual:` comment on the line ABOVE a pattern locks that pattern in its group. The comment text after `manual:` is a human-readable reason (required, for documentation).

### `# manual-group:` — Lock an Entire Group

```yaml
# manual-group: custom group for internal packages
custom-internal:
  patterns:
    - "@abinnovision/core-*"
    - "some-special-package"
```

The `# manual-group:` comment on the line ABOVE a group name locks the entire group. No patterns are added or removed. The comment text is a human-readable reason.

### Rules

- Comments without `manual:` or `manual-group:` prefix are informational and may be removed on regeneration
- The reason text after the prefix is required — forces authors to document WHY something is manually pinned
- Manual patterns can exist in auto-managed groups (mixed mode)

## Output Format

Follow the existing file's style. If creating fresh, use this structure:

```yaml
# NOTE: Auto-generated by generate-dependabot-config command.
# Lines marked with "# manual:" or "# manual-group:" are preserved across regenerations.
version: 2
updates:
  - package-ecosystem: npm
    directory: /
    versioning-strategy: increase
    open-pull-requests-limit: 15
    schedule:
      interval: weekly
      day: monday
      time: "13:00"
      timezone: Europe/Berlin
    labels:
      - dependencies
    commit-message:
      prefix: "fix(deps)"
      prefix-development: "build(deps/dev)"
    ignore:
      - dependency-name: "*"
        update-types: ["version-update:semver-patch"]

    groups:
      # Groups are sorted alphabetically.
      # Use "# manual:" above a pattern to lock it.
      # Use "# manual-group:" above a group to lock the entire group.
      react-related:
        patterns:
          - "react"
          - "react-dom"
          - "@types/react"
          - "@types/react-dom"
```

## Reference

For the full list of Dependabot configuration options (ecosystems, schedules, groups, ignore rules, versioning strategies, etc.), consult the official documentation: https://docs.github.com/en/code-security/reference/supply-chain-security/dependabot-options-reference

## Common Mistakes

- **Over-grouping**: Don't create a catch-all "misc" group. Individual PRs for one-off deps are better than a mega-group that updates 15 unrelated packages together.
- **Ignoring existing groups**: Always read the existing config first. Existing groupings represent deliberate choices — build on them, don't replace them.
- **Org-scope grouping**: Don't group `@abinnovision/*` together. These packages serve different concerns (eslint configs, nestjs libs, prettier configs) and should be classified by function.
- **Duplicate patterns**: `@vitejs/*` appeared in both `vitest-related` and `vite-related` in some configs. Each pattern should appear in exactly one group.
